Why This Works

なぜこのDEXは成功するのか。既存プレイヤーが解決できていない3つの構造的問題に、正しいタイミングで正しい解を提示する。

Core Thesis

DeFiの取引量は年間$1T超。しかし年間$10B以上がMEVとして搾取され、UXの悪さからユーザーの90%以上がCEXに留まっている。 この2つの問題は同じ根本原因 — 「注文の順序依存性」と「チェーンの分断」 — から発生している。 インテントベース + 暗号化バッチオークションはこれを構造的に解決する唯一のアーキテクチャであり、今まさにインフラ（ZK証明、AA、クロスチェーン）が実用段階に達した。

Reason 1

既存DEXのMEV問題は設計レベルで壊れている

Uniswapは注文が公開メモリプールを通過する。この設計ではフロントランニングは原理的に防げない。Hookやプライベートメモリプールは「パッチ」であり根本解決ではない。

我々の解:
閾値暗号で注文を秘匿 → バッチ確定まで誰も中身を見れない → フロントランニングが物理的に不可能。パッチではなくアーキテクチャで解決。
Reason 2

流動性の分断が「勝者総取り」の機会を生む

Ethereum、Solana、L2 — 流動性は20以上のチェーンに分散。ユーザーは手動でブリッジし、チェーンごとに別のDEXを使う。統一された体験を提供するプレイヤーはまだいない。

我々の解:
インテントレイヤーがチェーンを抽象化。ユーザーは「ETHをUSDCに」と言うだけ。ソルバーが全チェーンの流動性から最適ルートを発見。チェーンを意識させない。
Reason 3

AIが開発コストの常識を破壊する

従来、このレベルのプロトコルには50名以上のエンジニアと$30M以上の開発費が必要だった。AI（Claude/Cursor）の活用で、15名のチームが同等以上の品質を半分の期間で実現できる。

我々の解:
AI駆動開発で89,000 LOCを6ヶ月で実装。浮いた$12Mを流動性とセキュリティに全額投入。資金効率で既存チームを圧倒する。

#4 Intent Engine

ユーザーは「意図」だけ伝える。最適な実行はプロトコルが保証する。

1

Intent Submission

ユーザーが「1 ETHを最良価格でUSDCに交換」というインテントを署名して送信。EIP-712準拠の型付きデータ。

2

Threshold Encryption

インテントは閾値暗号で暗号化。バッチ確定まで誰も内容を閲覧できない → フロントランニング不可能。

3

Solver Auction (Dutch)

ソルバーが「ブラインド入札」で最良執行ルートを競争。CEX流動性、プライベートマーケットメーカーも参加可能。

4

Batch Decryption & Matching

バッチ期間（~250ms）終了後に一括復号。同一価格で約定（バッチオークション）。MEVは構造的に排除。

5

On-chain Settlement

約定結果をZK証明付きでオンチェーン確定。ユーザーのウォレットにトークンが直接到着。

Intent Schema

struct Intent { // 基本フィールド sender: Address, // ユーザーのウォレットアドレス sell_token: Token, // 売却トークン (e.g., WETH) sell_amount: U256, // 売却数量 (wei) buy_token: Token, // 購入トークン (e.g., USDC) min_buy: U256, // 最低購入量 (スリッページ保護) // 制約 deadline: u64, // UNIX timestamp 期限 src_chain: ChainId, // ソースチェーン dst_chain: Option<ChainId>, // 宛先チェーン (クロスチェーン時) partial_fill: bool, // 部分約定許可 // 署名 nonce: U256, signature: Signature, // EIP-712 typed data signature }

#5 MEV Protection

年間$10B以上が搾取されるMEV問題を構造的に解決

🔒 暗号化メモリプール

閾値暗号（t-of-n）で注文内容を秘匿。バリデータの2/3が復号に参加しない限り、注文内容は不明。サーチャーは何も見えない。

// Threshold BLS encryption let encrypted = tlock::encrypt( &intent_bytes, &committee_public_key, round_number + 1, // 次のバッチで復号 );

⚖ バッチオークション

250msバッチ内の全注文を同一価格で約定。注文の順序に意味がない → 先行者利益ゼロ。MEVの根本原因（順序依存性）を除去。

// Batch auction flow // t=0ms: バッチ開始、インテント受付 // t=250ms: バッチ締切、一括復号 // t=260ms: 均一価格計算 (CDA → UCP) // t=270ms: 全注文を同一価格で約定

💰 MEVリベート

それでも発生する残余MEV（統計的アービトラージ等）は80%をユーザーに還元。残り20%はプロトコル収益。

#6 Smart Contracts

最小限のオンチェーンロジック、最大限の検証可能性

Settlement.sol

原子的スワップ実行。ソルバーのソリューションを検証し、トークンを移動。Permit2統合でapprove不要。

function settle( Intent[] intents, Solution solution, bytes proof ) external { require(verifier.verify(proof)); for (intent in intents) { permit2.transferFrom(...); } }

Vault.sol

流動性プール管理。集中流動性ポジション（Uniswap V3互換tick範囲）。LPトークンはERC-721（NFT）。

Governance.sol

veTokenモデル。ロック期間に応じて投票力が増加。手数料分配、パラメータ変更、緊急停止の権限。

BridgeReceiver.sol

クロスチェーンメッセージの受信・検証。LayerZero / Hyperlane / Wormhole マルチブリッジ対応。

#8 Security Model

15億円のセキュリティ予算 — 1件のハックで全てが終わる

🔎 多層監査

3社以上の独立監査。Trail of Bits（低レベル）、OpenZeppelin（コントラクト）、Spearbit（経済モデル）。各モジュールを異なるチームが監査。

📊 フォーマル検証

Certora / Halmos でコントラクトの数学的正当性を証明。「スワップ後にユーザー資産が減少しない」等の不変条件を網羅。

🎯 バグバウンティ

Immunefi上で最大$10Mの報奨金。Critical（資金喪失）= $10M、High（プロトコル停止）= $1M。業界最高水準。

🚨 緊急対応

Circuit Breaker: 異常な価格変動/大量引出を自動検知。Guardian Multisig (5/9) でプロトコル一時停止。30日後にガバナンスへ権限移譲。
